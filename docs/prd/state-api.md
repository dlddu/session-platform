# PRD: 세션 상태 일관성 & Read/Write API

> 대상 요구사항: ④ Redis 기반 atomic operation, ⑥ 세션 단위 read/write API (상태별 분기), ⑦ 세션 간 자유 전환

## 달성 가치
- **V3 끊김 없는 세션 연속성** — 상태와 무관하게 read/write가 동작
- **V4 자유로운 멀티세션 전환** — 세션 간 이동 보장
- **V5 일관된 세션 상태** — Redis atomic operation으로 상태 일관성 확보

## Acceptance Criteria

### AC-C1: Redis 기반 atomic 상태 전이
- **설명**: 세션 상태 전이(active ↔ idle ↔ snapshot)와 세션 점유는 Redis를 통한 atomic operation으로 처리된다. 동일 세션에 대한 동시 요청(예: 복원과 스냅샷이 동시 발생)에서도 단일 전이만 성공하며 상태가 깨지지 않는다.
- **달성 가치**: V5
- **검증 방법**: 동일 세션에 복원/스냅샷/전환 요청을 동시 다발로 발생시켜, 최종 상태가 항상 유효한 단일 상태로 수렴하고 중복 pod 기동·이중 스냅샷이 발생하지 않음을 확인한다.

### AC-C2: Read API 상태별 분기
- **설명**: 세션 단위 Read API는 대상 세션의 상태에 따라 분기 처리한다.
  - `active`: pod에서 직접 읽어 즉시 응답
  - `idle`: 유휴 pod에서 읽기 *(또는 active 승격 후 읽기 — 정책 확정 필요)*
  - `snapshot`: CRIU 복원 후 읽기 *(또는 스냅샷 메타데이터만 응답 — 정책 확정 필요)*
- **달성 가치**: V3, V4
- **검증 방법**: 동일한 read 요청을 세 상태의 세션에 각각 호출하여, 상태별 지정 경로로 처리되고 올바른 결과를 반환함을 확인한다.

### AC-C3: Write API 상태별 분기
- **설명**: 세션 단위 Write API도 상태별로 분기한다. `snapshot`/`idle` 세션에 write가 들어오면 복원·승격 후 write를 적용하는 등 상태에 맞는 경로로 처리한다. *(snapshot 상태 write를 허용할지/거부할지 정책 확정 필요)*
- **달성 가치**: V3, V5
- **검증 방법**: 각 상태의 세션에 write 요청 시 데이터가 일관되게 반영되고 상태 전이가 atomic하게 일어남을 확인한다.

### AC-C4: 세션 간 자유 전환
- **설명**: 사용자는 보유한 여러 세션 사이를 자유롭게 전환할 수 있다. 전환 대상이 `snapshot`이면 복원하여 `active`로, 이미 `active`면 그대로 접근시킨다. 전환은 세션 격리(AC-A2)를 깨지 않는다.
- **달성 가치**: V4, V3
- **검증 방법**: 여러 세션을 오가며 전환 시 매번 대상 세션이 올바른 상태로 활성화되고, 이전 세션의 상태가 보존됨을 확인한다.
